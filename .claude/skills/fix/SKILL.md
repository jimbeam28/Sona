# 全量问题修复 (fix)

循环实现 `docs/dev/fix-status.json` 中所有 `impl_status == "pending"` 的修复，每个修复遵循 fix-ex 工作流，完成后记录日志并 git 提交推送。

---

## 循环前：环境检查

1. 确认当前目录是 git 仓库，有远程仓库 origin 且可推送（`git remote -v`）。
   - 如未配置远程仓库，提示用户先配置。
2. 确认 `docs/dev/fix-status.json` 存在。
3. 确认 `docs/dev/dev_log.md` 存在，如不存在则创建空文件。

---

## 循环体（每轮处理一个修复）

### 步骤 A：选择修复

读取 `docs/dev/fix-status.json`，找到所有 `impl_status == "pending"` 的修复。

选择策略：
- 按 `implementation_order` 数组的顺序遍历（step 1 → step 2 → ... → step 8）
- 每个 step 内按 `tasks` 数组顺序遍历（如 A-5 → A-6）
- 跳过 `impl_status == "done"` 的修复
- 检查 `dependencies`：如果依赖项中还有未完成（`impl_status != "done"`）的，跳过该修复
- 取第一个符合条件的 pending 修复

如果所有修复 `impl_status` 都已不是 `pending`，报告"所有修复已完成"并结束循环。

**向用户报告：** 本轮选中 `[编号] - [名称]`，优先级，关联问题，涉及文件。

### 步骤 B：执行 fix-ex 工作流

对此修复执行 fix-ex 技能定义的完整流程：

1. **解析修复配置**：读取 fix-status.json 和 fix.md 中对应的修复方案。
2. **实现修复**（新 Agent）：严格按修复文档实现代码。
3. **验证修复**（新 Agent）：对照修复文档逐项检查，有偏差直接修复。

详细规范见 fix-ex skill 的第1-3步。

### 步骤 C：追加修复日志

将本轮修复的摘要追加到 `docs/dev/dev_log.md`。如果文件不存在，先创建。

日志格式：

```markdown
---

## [YYYY-MM-DD HH:MM] [编号] - [名称]

**优先级**: P0/P1/P2
**关联问题**: [问题编号列表]
**状态**: ✅ 成功 / ⚠️ 部分完成（说明原因）

### 修改文件
- `lib/path/to/file1.dart` — 简要说明
- `lib/path/to/file2.dart` — 简要说明

### 验证结果
- 通过: X / 总计: Y（work_items 检查项）
- （如有未通过项，列出具体原因）

### 备注
（如有异常、注意事项、手动修复内容等）
```

### 步骤 D：更新状态并提交

1. 更新 `docs/dev/fix-status.json` 中该修复的 `impl_status`：
   - 成功 → `"done"`
   - 失败 → 保持 `"pending"`（由中断处理逻辑处理）

2. Git 提交并推送：
   ```bash
   git add -A
   git commit -m "fix([批次]): 修复 [编号] - [名称]"
   git push
   ```
   - 如果 push 失败（如冲突），报告错误并暂停，等待用户处理后再继续。

### 步骤 E：继续下一个

回到步骤 A，选择下一个 pending 修复。

---

## 循环结束

所有修复 `impl_status == "done"` 后，汇报总览：

```
═══════════════════════════════════
  全量修复完成
═══════════════════════════════════
  总修复数:    N
  成功修复:    N
  Git 提交数:  N
  按优先级:    P0: N, P1: N, P2: N
═══════════════════════════════════
```

---

## 中断处理

如果某轮循环中步骤 B（fix-ex 流程）失败：
1. 记录失败原因到日志。
2. 将该修复 `impl_status` 保持为 `"pending"` 并附带备注。
3. 检查是否有其他修复依赖此修复，如有则跳过那些修复。
4. 继续处理下一个修复，不中断整体循环。
5. 循环结束后，列出所有失败的修复供用户检查。

如果 git push 失败：
1. 暂停循环。
2. 提示用户解决推送问题。
3. 问题解决后继续循环。

## 依赖处理

修复之间存在依赖关系（如 A-1 依赖 A-2/A-3/A-4/A-5）。选择修复时：
- 优先选择无依赖或依赖项已全部完成的修复
- 如果某修复的依赖项尚未完成，自动跳过该修复
- 当被跳过的修复的依赖项在后续轮次中完成后，下一轮自动选中该修复
