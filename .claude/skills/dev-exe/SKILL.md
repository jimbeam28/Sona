# 代码实现 (dev-exe)

读取 `docs/dev/dev-status.json`，按 `dev-plan.md` 中对应章节实现代码。

## 输入

可选的**条目编号**（如 `CON-01`、`A-1`、`I-4`）：
- **有编号**：只处理指定条目（单条模式）
- **无编号**：循环处理所有 pending 条目（批量模式）

## 前置：读取上下文

- `docs/dev/dev-status.json` — 进度跟踪
- `docs/dev/dev-plan.md` — 开发计划

---

## 单条模式工作流

### 第1步：解析条目配置

1. 从 `dev-status.json` 的 `items` 中找到目标条目
2. 提取：`name`、`plan_section`、`impl_status`、`dependencies`
3. 根据 `plan_section` 读取 `dev-plan.md` 中对应章节，理解完整实现要求
4. 检查 `dependencies`（含传递依赖）：
   - 递归检查：如果 A 依赖 B、B 依赖 C，则 A 的完整依赖链为 [B, C]
   - 如果依赖链中有 `impl_status` 不是 `"done"` 的条目，**停止**，提示用户先完成依赖项
   - 提示信息列出所有未完成的依赖项（含传递依赖），而非仅直接依赖
5. 检查当前状态：
   - `blocked`：**停止**，提示用户该条目已被阻塞（失败次数过多），需手动介入
   - `failed`：继续执行，但在确认时提示用户当前重试次数

**向用户确认：**
- 条目编号和名称
- 实现要点
- 涉及文件
- 依赖项状态

### 第1.5步：收集代码上下文

在启动实现 Agent 前，先收集涉及文件的关键代码片段：

1. 读取计划中"涉及文件"列出的每个文件
2. 读取"代码锚点"中引用的文件，提取锚点位置 ±30 行的代码上下文
3. 读取相关联的文件（如修改 Provider 时，读取其依赖的 Service/DAO 接口）
4. 汇总为结构化的代码上下文，供实现 Agent 使用

### 第2步：实现代码与测试

启动新的 Agent（general-purpose），严格按 `dev-plan.md` 中该条目的实现要点编写代码，同时实现测试用例。

Agent prompt 必须包含：
- `dev-plan.md` 中该条目的完整内容（实现要点、代码锚点、涉及文件、测试用例、验收标准）
- **第1.5步收集的代码上下文**（涉及文件的关键代码片段，含文件路径和行号）
- 架构约束（分层：UI → Provider → Service → Data）
- 测试工具配置（mockito、sqflite_ffi、fake_async、ProviderContainer 等）
- **强制约束：严格按计划实现，不得自行发挥。不得跳过任何实现要点。每个测试用例都必须有对应的测试代码。**
- **代码定位：优先使用计划中的代码锚点和上下文中的代码片段定位修改位置，不要自行搜索。**

Agent 输出：
- 创建/修改的文件列表（代码 + 测试）
- 与实现要点和测试用例的对应关系
- **实现决策记录**：对计划中有多种实现方式的要点，记录选择了哪种方案及原因

### 第3步：验证实现并运行测试

启动新的 Agent（general-purpose），对照 `dev-plan.md` 逐项验证实现，运行测试并修复直至全部通过。

Agent prompt 必须包含：
- `dev-plan.md` 中该条目的完整实现要点、代码锚点、测试用例和验收标准
- 第2步输出的文件列表
- 第2步输出的**实现决策记录**（了解实现 Agent 的设计选择，避免将合理决策误判为错误）
- **验证标准：逐项核对每个实现要点是否已正确完成，每个测试用例是否已覆盖，每条验收标准是否满足**

Agent 需要：
1. 读取每个实现文件的代码，逐项核对实现要点
2. 对照验收标准逐项检查：
   - 所有测试用例通过
   - 无新增 warning
   - 实现与计划要点一一对应（无遗漏、无自行发挥）
   - Bug 修复：根因已消除，非症状掩盖
3. 发现偏差或遗漏直接修复
4. 运行 `flutter test`，分析失败原因
5. 如果代码不符合计划 → 修改代码
6. 如果测试与用例描述不一致 → 修改测试
7. 重复直到全部通过
8. 报告：验证结果、测试通过数/总数、验收标准通过情况、修复了哪些问题

### 第4步：静态分析与全量测试

**在本会话中直接执行（不使用 Agent）：**

1. 运行 `flutter analyze`，确认 0 issues
2. 运行 `dart format --set-exit-if-changed lib test`，确认无格式变更
3. 运行 `flutter test`，确认全部通过（回归检查）
4. 如有失败，直接修复，重复直到全部通过

**质量门禁：** 第4步必须全部通过，否则视为未完成。

## 批量模式工作流

### 循环前：环境检查

1. 确认 git 仓库有远程 origin 且可推送
2. 确认 `docs/dev/dev-status.json` 存在
3. 确认 `docs/dev/dev_log.md` 存在（不存在则创建）

### 循环体（每轮处理一个条目）

**步骤 A：选择条目**

读取 `dev-status.json`，按 `order` 数组顺序找第一个 `impl_status == "pending"` 的条目。跳过依赖未完成的条目，跳过 `impl_status == "blocked"` 的条目。

如果无 pending 条目，报告完成并结束。

选择后输出进度：
```
─── 进度 [N/M] ───
  当前: [编号] - [名称]
  已完成: N 个 | 剩余: M 个 | 失败: K 个 | 阻塞: J 个
──────────────────
```

**步骤 B：执行单条模式工作流**

对该条目执行第1-4步。

**步骤 C：追加日志**

```markdown
---

## [YYYY-MM-DD HH:MM] [编号] - [名称]

**状态**: ✅ 成功 / ⚠️ 失败（原因）

### 修改文件
- `lib/path/to/file.dart` — 说明

### 测试结果
- 通过: X / 总计: Y
```

追加到 `docs/dev/dev_log.md`。

**步骤 D：更新状态并提交**

1. 更新 `dev-status.json`：
   - 成功：`impl_status = "done"`, `test_status = "passed"`
   - 失败：`impl_status = "failed"`，`retry_count += 1`，`last_error = "失败原因摘要"`
   - 失败且 `retry_count >= 3`：`impl_status = "blocked"`，不再自动重试
2. Git 提交推送：
   ```bash
   git add -A
   git commit -m "feat: [编号] - [名称]"  # 或 fix: [编号] - [名称]
   # 提交信息体包含修改文件列表和测试结果，便于 git log 回溯
   git push
   ```
   提交信息格式：
   ```
   feat: [编号] - [名称]

   - 修改文件: lib/path/a.dart, lib/path/b.dart
   - 测试: X/Y 通过
   - 静态分析: 0 issues
   ```

**步骤 E：继续下一个**

---

## 中断处理

- 某条目失败 → 记录日志，标记 `failed`（retry_count+1, last_error），继续下一个
- 某条目失败 3 次 → 标记 `blocked`，batch 模式跳过，汇报中单独列出
- git push 失败 → 暂停，等待用户处理
- 依赖未完成 → 跳过该条目
